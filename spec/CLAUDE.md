## 1. 絶対制約 — 共通通信ルール（全画面共通。画面ごとのシークエンス図はこの3行で代替する）

1. 外部正本（freee／HubSpot／board／GreatSign）が正。ローカルに独自テーブルを作らず、表示のたび正本から取得する（キャッシュは表示用途のみ・書き戻し禁止）。
2. 正本への書き込みは、screens/ 各対応表で W-xx として定義された操作のみ。楽観更新禁止。成功応答を受けてから画面を更新する。
3. 状態遷移の可否は domain/*.state.mmd を唯一の正とする。コードに遷移表を持つ場合、生成元コメントで同ファイルを指すこと。

## 2. ナビゲーション共通ルール（構造の正は screens.yaml）

1. **主ナビとアドミンモードの分離**：「設定」はページではなくモード。押すと左サイドバーがアドミン用（ST-xx 群）に差し替わり、閉じると主ナビに戻る。日常の閲覧・操作（主ナビ）と、たまに行う編集・設定（ST-xx）を混在させない。権限のないユーザーには設定ボタン自体を出さない。
2. **アラート整合**：サイドバーのバッジ数は、必ず遷移先画面内の対応要素（カード・行）と1対1で紐づく。バッジをクリックすると該当要素へスクロール/ハイライトする。内訳が画面のどこにも存在しないバッジを表示してはならない。
3. **文書は単体で開かない**：文書は会社詳細・案件詳細の中の文脈で参照する。テンプレート管理のみ ST-03。
4. **タブの全数有効**：画面にタブを置く場合、全タブが機能するまでその画面は Gate 1 を通過できない（押せないボタンを本番系画面に残さない）。

## 3. 品質定規（CDO監査。全画面、生成後にこの物差しで自己監査してから完了とする）

| # | 定規 | 合格基準 |
|---|------|---------|
| Q1 | コントラスト | 画面の最重要数値が補足テキストの約3倍の大きさ＋状態色で際立つ。1画面1メッセージ。全部を目立たせない |
| Q2 | 近接 | 関連情報は1カードに集約、カード間余白はカード内余白より大きい |
| Q3 | 整列 | 全カード・全要素が共通グリッドに揃う（bentoのズレ＝Q3違反） |
| Q4 | 反復 | 同役割の要素（見出し・カード・バッジ・ボタン）は全画面で同一トークン（色・フォント・余白・角丸）を使う |
| Q5 | 情報の形 | 状態・比較・割合・推移を文章で説明しない。バッジ・横棒・進捗バー・折れ線に置換。文章は掴んだ後に読む補足として小さく淡く |
| Q6 | 視線誘導 | 最重要情報は左上。大局→詳細の順に流れる |
| Q7 | 装飾排除 | フォント2種以内。色はトークン定義内のみ。装飾目的で色を増やさない |
| Q8 | 露出文言 | 内部の仮名・ダミー（例:「PGプラット」）を表に出さない。サンプル値は薄字＋「サンプル」明記（SAMPLE警告と同思想） |
| Q9 | アラート整合 | §2-2 を満たす（バッジ⇄画面内要素の1対1） |

**デザイントークン（正はここ。各画面に複製しない）**：主色＝Initial State ブランド1色＋濃淡／状態色＝プラス・マイナス・注意・良好の4種／余白スケール1系統／本文1書体＋見出し1書体。色だけに情報を頼らず、色＋ラベル/アイコンで二重に伝える。

## 4. 実装の順序

1. 対象画面の screens.yaml エントリと screens/（カンプ＋対応表）、書き込みがある画面は .feature を読む。
2. 曖昧・未定義に出会ったら実装せず assumption-ledger.md に前提を1行追記し、その前提に沿って続行する。推測を黙って埋めない。
3. カンプの data-xx 属性と対応表の P-xx／W-xx を突き合わせ、対応表にない項目は静的ラベルとして実装する。
